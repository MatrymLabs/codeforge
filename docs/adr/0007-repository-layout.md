# ADR-0007: Repository layout and boundaries

**Status:** accepted (revisable - a structure choice, revisited as the ship grows)

## Context

CodeForge grew organically into a large flagship: ~246 files under `parts/`, a matching
`tests/` twin tree, generated evidence, game content, docs, and delivery mechanics. A research
review ("Repository & File Structure Best Practices: Python and Multilanguage Stacks") asked a
fair question: does the layout match professional convention, and where should each kind of
file live? This ADR is the answer of record, so the structure is **defensible** rather than
merely inherited. It fixes the boundaries an interviewer, a future teammate, or an AI assistant
should navigate by, and it records the one migration we deliberately did **not** do.

## Decision

CodeForge is a **single-package, monorepo-style flagship** with a flat importable core, not a
`src/`-layout project and not (yet) a split of many repos. The boundaries:

### 1. Where production code lives

- **`parts/`** - the importable package: the engine's parts (one job + a `CARD:` line + a
  test twin each) and the Hardware Store's reusable parts. Subpackages group cohesive
  subsystems (`parts.evolution`, `parts.stewardship`, `parts.hubble`).
- **`forge.py`** (repo root) - the engine tick, `handle_command(session, signal) -> str`. It
  is the only door (ADR-0001) and stays at the root as a top-level `py-module`, exactly where
  the packaging config and the CLI entry points expect it.
- **`parts/cli.py`** - the console entry points (`codeforge`, `spark`), declared in
  `pyproject.toml [project.scripts]`. These verb strings are a frozen public contract.

### 2. Where each other kind of file belongs (the map)

| Concern | Home | Tracked? |
|---------|------|----------|
| Production code | `parts/`, `forge.py` | yes |
| Tests (unit + property + fuzz) | `tests/` (twin per part) | yes |
| End-to-end tests (separate conftest) | `e2e/` | yes |
| Game content (World Package **source**) | `seeds/`, `seed_templates/` | yes |
| Hardware Store catalog | `catalog/` | yes |
| Classification Registry | `registry/` | yes |
| Docs, ADRs, architecture, pattern docs | `docs/` (`docs/adr/`, `docs/hardware_store/`) | yes |
| Performance evidence | `benchmarks/` | yes |
| Automation scripts | `scripts/` | yes |
| DB migrations | `migrations/` + `alembic.ini` | yes |
| Career / pioneer data | `data/` | yes |
| Build/tool config (single source of truth) | `pyproject.toml` | yes |
| Deploy | `Dockerfile`, `docker-compose.yml`, `render.yaml` | yes |
| Generated evidence | `reports/` (only `reports/README.md` kept) | **no** (`reports/*` ignored) |
| Security evidence artifacts | `security-evidence/` | yes (dated, reproducible) |
| Runtime state | `codeforge.db`, `save.json`, `characters.json` | **no** (gitignored) |

### 3. Generated vs maintained

Generated artifacts are **git-ignored but reproducible from the recorded commit**: `reports/*`
(regenerated by `make patch`), the SBOM, coverage, caches, and all runtime state. Maintained
artifacts (code, tests, seeds, docs, the registry, the catalog) are tracked. A file is never
both: no generated output is hand-edited in place.

### 4. Flagship now vs independent repo later (question 15)

CodeForge stays a **single public flagship** while its parts remain tightly coupled to the
engine and the proving-ground game. A part earns its own repository only when it is (a) reused
outside CodeForge, (b) stable, and (c) carrying its own release cadence - none are yet. The
industry/compliance tooling (`ai-log-triage`, `federal-guidance-library`) already lives in
**separate private repos** by the shop-window / back-room strategy; that split is deliberate and
stands. Future JS/TS clients and infrastructure-as-code, when they exist, get their own
top-level directory (`frontend/`, `infra/`) or their own repo - see `docs/repo_templates/`.

## The migration we did NOT do: `src/` layout

The research's headline Python recommendation is a `src/` layout. **We decline it**, with
evidence:

1. **The benefit does not apply here.** `src/`'s purpose is to stop you importing an
   *uninstalled* copy of your package. CodeForge is already installed (editable; the CLI entry
   points and `codeforge.egg-info` prove it), so the import-the-wrong-copy hazard `src/` guards
   against is already closed a different way.
2. **The blast radius is the repo's worst-case failure mode at scale.** Moving `parts/` to
   `src/parts/` and `forge.py` to `src/forge.py` would rewrite `from parts.X` imports across
   **~246 source + ~240 test files**, plus registry `file:` paths, catalog `source:` paths,
   every PartManifest, the completeness gate, the packaging config, and docs. This repo has a
   documented scar here: *"a 12-file rename is a refactoring that rewrites imports across many
   files."* This would be that, at ~50x scale.
3. **No measured gain.** Per the optimization ethos, a change with enormous blast radius and no
   evidence of functional benefit is a `rejected` optimization, not an improvement. A prior
   review reached the same non-recommendation.

If a real trigger appears (publishing `parts` to PyPI as an independent library, or a genuine
uninstalled-import bug), this decision is revisited on its merits - not before.

## Template library (structures CodeForge does not need yet)

Per the research and the architecture prompt, we **do not create empty folders to look mature.**
The future archetypes (full-stack Python + JS front end, infrastructure-as-code, a standalone
library package, a data-science repo) live as **reference templates** in
`docs/repo_templates/`, honestly labeled *planned, not present*. When CodeForge actually grows a
front end or IaC, the template is the starting point - and this ADR gets a status update.

## Consequences

- **Navigability:** every kind of file has one documented home; the map above is the single
  answer to "where does this go?"
- **Defensibility:** the layout is now a recorded decision an interviewer can read, including
  the honest reason a conventional recommendation (`src/`) was declined.
- **Low churn:** no imports move, no public contract changes, no risk taken for cosmetics.
- **The honest cost:** a flat `parts/` package is large. We accept that in exchange for zero
  migration risk and a subpackage grouping that keeps cohesion visible. If `parts/` ever
  becomes hard to navigate, sub-grouping (not a `src/` move) is the first lever.
